# git2go `/v38` 适配 libgit2 1.9.6 设计

## 目标

将当前基于 libgit2 1.5 的 git2go fork 升级为模块 `github.com/northhalf/git2go/v38`，支持 libgit2 1.9.x，并将 vendored 和 CI 版本固定为 1.9.6。

`v38.0.0` 需要完成两类工作：

1. 保持现有 git2go 导出 API 的源码兼容，修复 libgit2 1.5 到 1.9 的编译、ABI 和行为差异。
2. 暴露选定的 remote/network、worktree、commit 和 repository 新 API。

第一版只支持 Linux，验证 bundled static、bundled dynamic、system dynamic 和 system static 四种构建路径。

## 已确认的产品决策

- 发布新的 Go major `/v38`，不复用 `/v34` import path。
- 现有导出类型、函数和方法以源码兼容为优先。
- vendored libgit2 固定为官方 v1.9.6 标签提交 `26055f5af74ab1cf636d272e8a34315496d3f06f`。
- system linking 接受整个 libgit2 1.9.x，拒绝其他 minor 和 major。
- Go 最低版本为 1.25，CI 测试 Go 1.25 和 1.26。
- 保持现有 `GitError`、错误码判断和 callback 原始错误透传语义。
- 不支持实验性 SHA-256 仓库，继续使用固定 20 字节 `Oid`。
- checkout 跟随 libgit2 1.9 语义。`CheckoutOptions{}` 的零值 strategy 表示 safe checkout，不再表示 dry-run。
- 只以正确性、资源安全和跨链接模式一致性作为性能验收，不设置基准性能门槛。
- `v38.0.0` 增加明确列出的新 API，不机械绑定 libgit2 1.6 到 1.9 的全部新增符号。

## 非目标

- 不支持 macOS 或 Windows CI。
- 不支持 SHA-256 object database、SHA-256 OID 或双哈希仓库。
- 不重做 git2go 的错误模型。
- 不引入模拟 libgit2 1.5 ABI 的长期 C 兼容层。
- 不实现当前项目没有的自定义 config backend 或 allocator 抽象。
- 不承诺对 libgit2 1.8、1.10 或 2.x 的兼容。

## 依赖环境

本地 bundled 构建需要 Git submodule、CMake、`pkg-config` 和 C 编译器。Debian 或 Ubuntu 使用：

```sh
sudo apt-get install -y build-essential cmake pkg-config
git submodule update --init
```

system dynamic/static CI 不依赖 Linux 发行版中的 libgit2 包。CI 从官方 v1.9.6 源码构建并安装到隔离前缀，避免发行版包版本滞后。

## 模块与构建架构

### Go 模块

将 `go.mod` module path 改为：

```text
github.com/northhalf/git2go/v38
```

将 `go` directive 提升到 `1.25`。更新 `golang.org/x/crypto`、`golang.org/x/sys` 和 `github.com/google/shlex`，并运行 `go mod tidy`。依赖版本必须同时支持 Go 1.25 和 1.26。

### libgit2 submodule

将 `vendor/libgit2` 更新到 v1.9.6 的 tag commit。提交中必须包含 gitlink 变化，不复制 libgit2 源码到普通目录。

### 链接模式

保留现有三个 build-tag 文件：

- `Build_system_dynamic.go`: `!static`
- `Build_system_static.go`: `static && system_libgit2`
- `Build_bundled_static.go`: `static && !system_libgit2`

三个文件使用相同的编译期版本判断：major 必须为 1，minor 必须为 9。patch 不设上限。若 headers 启用了实验性 SHA-256，编译期直接拒绝该配置，防止固定 20 字节 `Oid` 与 C 结构布局不一致。

bundled dynamic 继续通过 `PKG_CONFIG_PATH` 和 `LD_LIBRARY_PATH` 指向仓库内构建产物，因此不需要新增 Go build tag。

### CMake 构建

更新 `script/build-libgit2.sh` 到 libgit2 1.9 支持的 CMake 参数：

- 启用线程支持。
- 关闭 libgit2 自身测试和 CLI。
- bundled 模式关闭原生 HTTPS 和 SSH，继续测试 `http.go`、`ssh.go` 和 `transport.go` 中的 Go managed transports。
- 明确关闭实验性 SHA-256。
- static 和 dynamic 使用独立 build/install 目录。
- 不为生产构建启用 `GIT_DEPRECATE_HARD`，因为 `/v38` 需要保留 deprecated `update_tips` callback 的源码兼容路径。

可以增加独立的 deprecated API 清单检查，但它不能禁用兼容路径或改变发布构建。

## libgit2 1.9 兼容设计

### API 与 ABI 审计

当前绑定直接引用约 567 个 `git_*` 符号和 428 个 `GIT_*` 常量。迁移必须重新编译所有 Go/cgo/C 代码，不能只更新 submodule 和版本宏。

重点审计：

- `wrapper.c` 中的函数指针和 callback signature。
- `odb.go`、`refdb.go`、`transport.go` 和 `credentials.go` 使用的 `git2/sys` 接口。
- `remote.go` 中 remote callback、fetch options 和 push options 的结构布局。
- 所有手写 options 结构字面量。
- 所有将 Go enum 直接转换为 C enum 的位置。

libgit2 1.x 保证标准 API 源码兼容，但 ABI 和 `git2/sys` 只在同一 minor 内稳定。所有构建产物必须针对 1.9.x 重新编译。

### Options 初始化

继续使用对应的 `git_*_options_init` 初始化 C options。将受影响的手写 `version:` 结构字面量改为以下模式：

1. 调用 libgit2 init 函数。
2. 检查 init 返回值。
3. 覆盖 Go 调用方显式设置的字段。
4. 在同一函数族中成对释放字符串数组、callback payload 和 nested options。

该模式保证 libgit2 1.9 新字段获得官方默认值。

### Checkout 行为

libgit2 1.9 将 `GIT_CHECKOUT_SAFE` 改为零值，并改变 `GIT_CHECKOUT_NONE` 的数值。`CheckoutStrategy` 常量直接跟随 1.9 C 常量，不维持旧数值映射。

结果：

- `CheckoutOptions{}` 执行 safe checkout。
- dry-run 调用方必须设置 `Strategy: CheckoutNone`。
- 使用具名常量的现有代码继续编译。
- 依赖 strategy 零值为 dry-run 的代码需要修改。

README 和迁移指南必须把该变化列为高风险行为变化，并提供修改前后的示例。

### Remote callbacks

保留现有 `UpdateTipsCallback`，并标记 deprecated。新增 `UpdateRefsCallback`，其参数包含 ref name、old OID、new OID 和 refspec。

callback 安装规则：

- 只设置 `UpdateTipsCallback` 时，安装 `update_tips`。
- 设置 `UpdateRefsCallback` 时，安装 `update_refs`。
- 两者都设置时，安装两者，由 libgit2 1.9 的 `update_refs` 优先规则决定实际调用。
- Go 文档明确说明不要依赖两个 callback 都执行。

两个 callback 都通过 `HandleList` 获取 Go payload，并沿用 `errorTarget` 和 `setCallbackError`。

### 公开结构扩展

`ConfigEntry` 新增：

- `BackendType ConfigBackendType`
- `OriginPath string`

`ConfigLevel` 新增 `ConfigLevelWorktree`。保持已有字段和数值由 C 常量映射。

`BlameHunk` 增加 libgit2 1.9 的 committer、summary 和 modified-line 信息。所有可空 C 指针在转换前检查 nil。

### 错误、线程和 Go 生命周期

- C 调用和 `MakeGitError` 必须位于同一次 `runtime.LockOSThread` 区间。
- 更新 `script/check-MakeGitError-thread-lock.go`，使其能在 Go 1.25/1.26 下分析现有模式。
- C 调用完成后保留必要的 `runtime.KeepAlive`。
- 新 C wrapper 使用现有 `Free`、finalizer、weak owner 和 invalid-state 规则。
- callback payload 必须在成功、libgit2 错误和 Go callback 错误三条路径中都能 untrack。
- 新 libgit2 错误码加入 `ErrorCode` 和 stringer 输出，不新增替代 `GitError` 的错误体系。

## `/v38.0.0` 新 API

### Remote 与网络

扩展 `FetchOptions`：

- `Depth int`
- `FollowRedirects RemoteRedirect`

新增常量：

- `FetchDepthFull`
- `FetchDepthUnshallow`
- redirect policy 常量

扩展 `PushOptions`：

- `FollowRedirects RemoteRedirect`
- `RemotePushOptions []string`

扩展 remote connect options，使其支持 redirect policy。现有 custom headers、proxy 和 callbacks 继续使用原有 populate/free 流程。

新增：

- `UpdateRefsCallback`
- `Remote.OidType()`，映射 `git_remote_oid_type`
- User-Agent get/set
- server connect timeout get/set
- server timeout get/set
- compiled feature backend 查询

timeout 的 Go API 使用 `time.Duration`。转换为 C 的毫秒整数时：

- 零值表示系统默认值。
- 负值返回参数错误。
- 超过 C `int` 毫秒范围时返回参数错误。

`Remote.OidType()` 只能在 remote 已连接后成功。返回 SHA-1 时正常返回；返回其他 object format 时返回 unsupported object format 错误。

### Worktree

新增 `worktree.go`，定义拥有 `*C.git_worktree` 的 `Worktree` wrapper。它遵循仓库现有资源模式，提供 finalizer，并让无效状态和重复 `Free` 的处理与其他 wrapper 保持一致。

Repository API：

- `Worktrees()`
- `LookupWorktree(name)`
- `OpenWorktree()`
- `AddWorktree(name, path, options)`

Worktree API：

- `Name()`
- `Path()`
- `Validate()`
- `Lock(reason)`
- `Unlock()`
- `IsLocked()`
- `IsPrunable(options)`
- `Prune(options)`
- `Free()`

Options：

- `WorktreeAddOptions`，包含 lock、checkout existing、reference 和现有 `CheckoutOptions`。
- `WorktreePruneOptions`，包含 prune flags。

per-worktree config 通过打开 worktree repository 后使用 `ConfigLevelWorktree` 访问。不增加 libgit2 不存在的专用 config API。

### Commit 与仓库状态

新增 `CommitCreateOptions`：

- `AllowEmpty bool`
- `Author *Signature`
- `Committer *Signature`
- `MessageEncoding string`

新增：

- `Repository.CreateCommitFromStage(message, options)`
- `Repository.CommitParents()`
- `Repository.OidType()`
- 从环境解析 author 和 committer signatures 的 wrapper

`CreateCommitFromStage` 将 libgit2 的 unchanged 返回码保留为对应 `GitError`。

`Repository.CommitParents()` 从 `git_commitarray` 获取 parent pointers。为了在调用 `git_commitarray_dispose` 后返回有效 Go wrappers，每个 commit 必须先 duplicate，再包装为拥有资源的 `Commit`。转换失败时释放已 duplicate 的 commits 和原始数组。

`Repository.OidType()` 在当前构建中应返回 SHA-1。任何其他类型返回 unsupported object format 错误。

## 测试设计

### 回归基线

第一阶段只更新构建和兼容代码，要求 bundled static 1.9.6 下现有测试全部恢复后，才开始增加新 API。

`TestMain` 继续执行：

- callback handle registry 泄漏检查。
- remote pointer registry 泄漏检查。
- 全局 `Shutdown`。

### 兼容测试

新增或更新测试：

- `CheckoutOptions{}` 为 safe checkout。
- `CheckoutNone` 为 dry-run。
- `ConfigEntry` 的 backend type、origin path 和 worktree level。
- `BlameHunk` 新字段及 nil signature。
- 旧 `UpdateTipsCallback`。
- 新 `UpdateRefsCallback`。
- 两个 update callback 同时设置时的 1.9 优先行为。
- callback 返回普通 Go error 和 `GitError` 的传播。
- 新错误码字符串生成结果。

### 新 API 测试

Remote/network：

- shallow fetch。
- deepen。
- unshallow。
- remote push options。
- redirect none/initial/all。
- User-Agent。
- connect timeout 和 server timeout 的零值、正常值、负值和溢出值。
- connected remote OID type。

Worktree：

- add、list、lookup、open。
- lock、unlock 和 lock reason。
- validate。
- prunable 判断和 prune。
- per-worktree config。
- checkout options 的传递和 callback 清理。

Commit/repository：

- staged commit。
- unchanged staged state。
- allow-empty commit。
- 默认和显式 author/committer。
- 普通状态下的 next commit parents。
- merge 状态下的多个 parents。
- repository OID type。

### 网络 fixture

发布 CI 不依赖公网。将现有外部 HTTPS clone 测试替换为进程内 HTTP/TLS fixture。保留并更新进程内 SSH server 测试。

每个 fixture 必须在测试结束时关闭 listener、连接、goroutine 和临时目录，避免 race job 挂起。

## CI 设计

GitHub Actions 使用受支持版本并固定 major 或 commit，不继续使用已经过时的 `actions/setup-go@v1` 和 `actions/checkout@v1`。

矩阵：

1. Go 1.25，bundled static，完整测试。
2. Go 1.26，bundled static，完整测试。
3. Go 1.26，bundled dynamic，完整测试。
4. Go 1.26，source-installed libgit2 1.9.6，system dynamic。
5. Go 1.26，source-installed libgit2 1.9.6，system static。
6. Go 1.26 race job，至少覆盖 callback、handle registry、remote registry 和本地 repository 操作。
7. generation/format/vet job。

每个 test job运行 `script/check-MakeGitError-thread-lock.go`，然后运行对应的 `go test --count=1` 命令。

generation job：

- 固定 stringer 版本，不使用 `@latest`。
- 运行 `make generate`。
- 运行 `gofmt` 检查。
- 运行 `go vet -tags static ./...`，复用 generation job 已构建的 bundled static libgit2。
- 运行 `git diff --quiet --exit-code`。

## 发布与迁移文档

发布 `v38.0.0` 前必须记录：

- 新 module/import path。
- Go 1.25 最低版本。
- libgit2 1.9.x 兼容范围和 vendored 1.9.6。
- checkout 零值从 dry-run 改为 safe checkout。
- `UpdateTipsCallback` deprecated，推荐 `UpdateRefsCallback`。
- SHA-256 仓库不支持。
- Linux-only CI 承诺。
- 新 remote、worktree 和 commit API。

发布 tag 必须与 `/v38` module path 匹配。不能复用当前仓库中 v34/v35 指向同一 commit 的历史做法。

## 风险与缓解

### Checkout 产生非预期写入

风险最高。迁移指南、测试和 release notes 都必须突出零值行为变化。示例代码使用具名 strategy。

### `git2/sys` ABI 变化

所有链接模式重新编译，system 模式不复用为 1.5 构建的二进制。CI 从源码构建 1.9.6。

### Callback payload 泄漏或 use-after-free

保留 `TestMain` registry 检查，新增 callback error path 测试和 race job。populate/free 函数成对审查。

### 新 options 默认值错误

统一调用 libgit2 init 函数，不复制 C 宏默认值到 Go。

### SHA-256 OID 长度不匹配

构建时关闭实验功能。repository 和 remote OID type API 在读取 OID 前拒绝非 SHA-1。

### 网络测试不稳定

发布门槛只使用进程内 HTTP/TLS/SSH fixture，不依赖公共 Git 服务。

## 验收标准

`v38.0.0` 可以发布时必须满足：

- `/v38` module 在 Go 1.25 和 1.26 下构建。
- vendored libgit2 为 v1.9.6。
- system linking 接受 1.9.x 并拒绝其他 minor。
- 四种 Linux 链接模式通过各自测试。
- 现有回归测试和新增 API 测试通过。
- race job 未发现 Go 数据竞争。
- callback 和 remote registries 在测试退出时为空。
- `go vet`、格式检查、线程锁检查和 generation check 通过。
- README 和迁移指南明确 checkout、Go 基线和 SHA-256 限制。
- 发布过程不依赖公网测试服务。
